    1.2. Project Management¶

    1.2.1. Basic principles¶

    Projects in CCP4 Cloud are designed to keep the track of structure solution process in such a way that it may be reviewed at any time later.

    Apart from straightforward cases, structure solution involves repeating certain actions and branching due to decision-making or exploration of a range of parameters. Correspondingly, project structure in CCP4 Cloud is a tree, rather than a linear (chronological) list of jobs.

    Projects Trees are built with a few basic operations, associated with the first five buttons in the vertical toolbar on the left-hand side of the screen:

    Note

    all buttons are duplicated by items in the dropdown context menu, which is displayed upon right-clicking on a job line in the Tree

    Add a job. The new job will be added as a child to one currently selected in the Tree. When this button is pressed, the new job is chosen from the list of tasks, which opens automatically. Repeat the addition of jobs to the same node will result in Project Tree branching.

    Note

    the context menu also offers the addition of a job with parameters set in the previous invocation of that type of job up the Project Tree. You may find this particularly useful in the model building and refinement stage when a number of similar-type jobs need to be run in multiple successions

    Clone job. The new job will be added as a sibling to one currently selected in the Tree, initialised with the same data and input parameters. Job dialog for the cloned job will be opened automatically, and you can change the input before running the job.

    Delete job. The currently selected job and all jobs found in the tree branch descending from it will be deleted after confirmation.

    Warning

    Once deleted, jobs cannot be restored.

    Stack/Unstack jobs. Excessively long project branches can be optimised (shortened) by stacking jobs in their linear sections (i.e., sections where each job has exactly one ancestor and one descendant). A typical example of such a section is given by the succession of model building and refinement jobs leading to a completed structure; these sections may reach tens and hundreds of jobs, which may be found inconvenient to work with. Stacking jobs are placed in a stack, which substitutes all of them in the branch. When necessary, the folder can be expanded back (unstacked).

    Stacking can be done in two ways:

    1. Select all jobs to be stacked and click on the button in the toolbar.

    Note

    unless selected jobs represent a continuous linear section of the project branch, the button is disabled

    Note

    it is possible to include a previously stacked folder in the selection, in which case it will be merged with the new one

    1. Select a job in a linear section of the project branch and click the button. The GUI will automatically identify boundaries of the linear section, and make lists of jobs, suitable for stacking, that are ancestors or descendants of the selected one. If one of the lists is empty, the other is used for stacking. If both lists are not empty, the user is asked which one, or both, should be used.

    Note

    automatically identified linear sections are constrained by either project branching or previously created stack folders

    For unstacking jobs, select the stack folder and click the button.

    Remarks are used for general annotation and can be placed in any place of the Projects. Remarks can be sticky (denoted by a pin dot in the icon: ) and non-sticky. Sticky remarks make a pair with their parent jobs and are moved and deleted synchronously with them. Unlike all other items in the Project Tree, remarks can be deleted without deleting their descendants.

    1.2.2. Managing the jobs¶

    The following buttons in the toolbar do not change the Project Tree:

    Open Job dialog for the currently selected job. Alternatively, double click the corresponding line in the Project Tree. Job dialogs are used for setting the job’s input data and parameters, and for running them.

    Stop the currently selected job. This button is active only when a job is running. Once stopped, jobs cannot be resumed or restarted. However, you can clone job that was stopped and run the clone.

    1.2.3. Branch highlighting¶

    In complex projects, it may be difficult to trace branches of interest. This may be helped by branch highlighting. Select a job and click the button in the toolbar (or the corresponding item in the context right-click menu). This will highlight all ancestors of the selected job to the project origin, and the descending branch which contains the most recent job. In order to clear the highlighting, click the button again.

    1.2.4. Refresh¶

    Nobody and nothing is perfect, and sometimes the Project Page may stop working for you due to intermittent connection errors or a hidden bug. Should this unlikely event happen, click the button to refresh the Project Page. simply reloads the project data and re-initialises internal page functions.

    Note

    this action is different from that of the browser’s Refresh button; the latter will log you out.

    1.2.5. Project development¶

    You can start developing your first Project right after creating it. In most cases, you should start with a task from the “Data Import” section of the Task List at the root of the Project Tree in order to import available data (reflection datasets, sequences, structure homologs, ligand descriptions , etc.), and then add computational jobs which use the imported data, as appropriate.

    You can create and run any number of jobs simultaneously, however, a child job cannot use its parent’s job output until the parent job is completed.

    CCP4 Cloud is not prescriptive about how you should grow and branch your Project Tree. It is highly advisable, though, to understand the principles of data management in CCP4 Cloud in order to avoid the most common mistakes and confusion. Please read the corresponding documentation.
